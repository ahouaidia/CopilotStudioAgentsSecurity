Why anatomy matters for security

Each component of a Copilot Studio agent represents a potential attack vector. The orchestrator can be manipulated through prompt injection, knowledge sources can leak sensitive data, actions can be abused for privilege escalation, and channels can be exploited for unauthorized access. Understanding the anatomy is the foundation for building a comprehensive threat model.

Top 10 Risks You Can Detect & Prevent

Common agent misconfigurations observed in real environments — and the controls to fix them. Source: Microsoft Security Blog

Sharing an agent with your entire organization or broad security groups exposes its capabilities without proper access boundaries. Users unfamiliar with the agent's purpose might unintentionally trigger sensitive actions, and threat actors with minimal access could use the agent as an entry point.

Mitigating Controls
  • Enable authentication (on by default)
  • Configure agent sharing limits in Managed Environments
  • Automatic security scan warnings at design-time and publish
Advanced Hunting Query
AI Agents – Organization or Multitenant Shared

Agents accessible without authentication create a significant exposure point. When publicly reachable, anyone with the link can use its capabilities. Topics, actions, or knowledge sources might unintentionally reveal internal information. This gap appears because authentication was deactivated for testing, left in its default state, or misunderstood as optional.

Mitigating Controls
  • Enforce agent authentication at the environment level
  • Automatic security scan warnings at design-time and publish
Advanced Hunting Query
AI Agents – No Authentication Required

Agents performing direct HTTP requests targeting non-standard ports, insecure schemes, or sensitive services bypass the governance, validation, throttling, and identity controls that connectors provide. Makers might copy a sample request, test an internal endpoint, or use HTTP actions for flexibility, leading to agents issuing unsecured calls.

Mitigating Controls
  • Apply data policies / advanced connector policies per environment
  • Communicate best practices in Maker welcome content
Advanced Hunting Queries
AI Agents – HTTP Requests to connector endpoints AI Agents – HTTP Requests to nonHTTPS endpoints AI Agents – HTTP Requests to nonstandard ports

When an agent uses generative orchestration to send email, the orchestrator determines the recipient and message content at runtime. In a successful indirect prompt injection attack, a threat actor could instruct the agent to send internal data to external recipients. Because email is an immediate outbound channel, any misconfiguration can lead to unmonitored data exposure.

Mitigating Controls
  • Implement Microsoft Defender Real-time Protection
  • Disable specific actions with connector action control
  • Use human-in-the-loop approvals for high-risk operations
Advanced Hunting Queries
AI Agents – Sending email to AI-controlled input values AI Agents – Sending email to external mailboxes

Unmonitored entry points often lack active ownership. A published but unused agent can still be called. A dormant maker-authenticated action might trigger elevated operations. Unused actions in classic orchestration can expose sensitive connectors if activated. These assets fall outside normal operational visibility while teams focus on active agents.

Mitigating Controls
  • Review agents in Power Platform Inventory
  • Quarantine agents flagged for deprecation
Advanced Hunting Queries
AI Agents – Published Dormant (30d) AI Agents – Unpublished Unmodified (30d) AI Agents – Unused Actions AI Agents – Dormant Author Authentication Connection

When agents use maker's personal authentication, they act on behalf of the creator rather than the end user. Every user inherits the maker's permissions. If those include access to sensitive data or high-impact connectors, the agent becomes a path for privilege escalation. MCP tools with maker credentials allow threat actors to trigger operations relying directly on the creator's identity.

Mitigating Controls
  • Restrict maker-provided credentials
  • Automatic security scan warnings at design-time and publish
Advanced Hunting Queries
AI Agents – Published Agents with Author Authentication AI Agents – MCP Tool with Maker Credentials

Clear-text secrets embedded in agent logic can be read, copied, or extracted by unintended users or automated systems. This often occurs when makers paste API keys, authentication tokens, or connection strings during development. These credentials bypass standard Key Vault policies, environment variable isolation, and are not rotated.

Mitigating Controls
  • Store secrets in Azure Key Vault, referencing as environment variables
  • Communicate best practices in Maker welcome content
  • Managed environments: Sharing Limits, Deployment Pipelines with gated extensions
Advanced Hunting Query
AI Agents – Hardcoded Credentials in Topics or Actions

MCP tools provide a powerful way to integrate with external systems, but if not actively maintained or reviewed, they can introduce undocumented access patterns. Unmonitored MCP tools might expose capabilities that exceed the agent's intended purpose, especially if they allow access to privileged operations or sensitive data sources.

Mitigating Controls
  • Manage available MCP tooling with Data Policies
  • Apply Advanced Connector Policies
  • Review Power Platform Inventory for dormant MCP integrations
Advanced Hunting Query
AI Agents – MCP Tool Configured

Instructions are the primary way to align a generative model with its intended purpose. Without them, the orchestrator lacks context to limit its output. A lack of guidance can cause agents to drift from expected behaviors, follow unexpected logic paths, or interact with connected systems in unintended ways.

Mitigating Controls
  • Communicate Best Practices for agent instructions
  • Azure Prompt Shield & RAI guardrails (on by default)
  • Implement Microsoft Defender Real-time Protection
Advanced Hunting Query
AI Agents – Published Generative Orchestration without Instructions

Orphaned agents whose owners are no longer with the organization or have deactivated accounts continue to run, interact with users, or access data without an accountable individual. They bypass standard review cycles and often contain outdated logic, deprecated connections, or sensitive access patterns that don't align with current requirements.

Mitigating Controls
  • Check inventory for stale agents
  • Quarantine agents to be decommissioned
  • Reassign ownership or delete orphaned agents
Advanced Hunting Query
AI Agents – Orphaned Agents with Disabled Owners

Mitigation Playbook

Five concrete steps to close common agent security gaps before they're exploited. Click any card to reveal the action checklist. Source: Microsoft Security Blog

1

Verify Intent & Ownership

Confirm every agent has a purpose and an accountable owner

Risk 1 Risk 2 Risk 5 Risk 10
  • Validate business justification for broad sharing, public access, or elevated permissions with the agent owner
  • Confirm agents without authentication are explicitly designed for public use and approved by policy
  • Review agent topics, actions, and knowledge sources for unintentional data exposure
  • Ensure every agent has an active, accountable owner — reassign or retire orphaned agents
  • Validate whether dormant agents, connections, or actions are still required
  • Perform periodic reviews and establish a clear organizational policy for agent creation
2

Reduce Exposure & Tighten Access

Limit who can reach the agent — and what the agent can reach

Risk 1 Risk 2 Risk 4
  • Restrict agent sharing to well-scoped, role-based security groups
  • Establish organizational policies defining when broad sharing or public access is allowed
  • Enforce full authentication by default — only allow unauthenticated access when explicitly approved
  • Restrict email actions to approved domains or hardcoded recipients
  • Avoid AI-controlled dynamic inputs for sensitive outbound actions
  • Perform periodic reviews of shared agents to ensure visibility remains appropriate
3

Enforce Strong Auth & Least Privilege

Agents must not inherit more privilege than necessary

Risk 6 Risk 7 Risk 8
  • Replace author authentication with user-based or system-based authentication
  • Review all tools running with delegated access and reconfigure those exposing sensitive services
  • Audit MCP tools that rely on creator credentials — remove or update if no longer required
  • Apply the principle of least privilege to all connectors, actions, and data access paths
4

Harden Orchestration & Dynamic Behavior

Generative agents need explicit guardrails to prevent unsafe behavior

Risk 3 Risk 4 Risk 9
  • Configure clear, well-structured instructions for generative orchestration
  • Avoid allowing the model to dynamically decide email recipients, external endpoints, or execution logic
  • Review HTTP Request actions — confirm endpoint, scheme, and port are required
  • Prefer official connectors over raw HTTP requests for authentication, governance, and logging
  • Enforce HTTPS and avoid nonstandard ports unless explicitly approved
5

Eliminate Dead Weight & Protect Secrets

Unused capabilities and embedded secrets quietly expand the attack surface

Risk 5 Risk 7 Risk 8 Risk 10
  • Remove or deactivate dormant agents, unpublished drafts, unused actions, and stale connections
  • Clean up outdated or unnecessary MCP tool configurations
  • Remove actions using author authentication in classic orchestration that are no longer referenced
  • Externalize all secrets to Azure Key Vault — reference via environment variables, never embed in agent logic
  • Treat agents as production assets and include them in regular lifecycle governance reviews

Threat Modeling

Mapping trust boundaries, data flows, and control points across the agent lifecycle to identify where threats emerge.

Agent Reference Data Flow
Common agent data flows start with user prompts entering the system as raw data. The orchestrator — whether a language model or logical code flow — matches prompts to appropriate response logic through iterative intent detection and response matching processes.
Threat Modeling Approach
Threat Modelling AI agents and solutions should be backed by the MITRE ATLAS© and OWASP® frameworks for identifying risks across all AI workloads.

Each step in the data flow represents a potential security boundary and control point where you must enforce trust, traceability, and transparency requirements to maintain system integrity and user confidence.
